### 重要ポイント（試験頻出）
✅ **Azure Monitor メトリクスの種類**
- Lead Time: 要求から完了までの総時間（MTTR: 平均復旧時間）
- Cycle Time: 作業開始から完了までの時間
- Velocity: スプリントで完了したバックログ数

✅ **Azure Monitor 通知**
- Action Group で通知先を管理
- Logic App で Teams などに通知

✅ **Dependency Agent**
- Service Map には Dependency Agent が必須
- Azure Monitor Agent (AMA) では代替不可

#### 1.5 カスタムテレメトリの追加

新しいコントローラーを作成 `Controllers/MetricsController.cs`:

```csharp
using Microsoft.ApplicationInsights;
using Microsoft.ApplicationInsights.DataContracts;
using Microsoft.AspNetCore.Mvc;

namespace WebApp.Controllers;

[ApiController]
[Route("api/[controller]")]
public class MetricsController : ControllerBase
{
    private readonly TelemetryClient _telemetryClient;
    private readonly ILogger<MetricsController> _logger;

    public MetricsController(TelemetryClient telemetryClient, ILogger<MetricsController> logger)
    {
        _telemetryClient = telemetryClient;
        _logger = logger;
    }

    [HttpGet("track")]
    public IActionResult TrackMetric()
    {
        // カスタムメトリクスを送信
        _telemetryClient.TrackMetric("CustomMetric", 42);
        
        // カスタムイベントを送信
        _telemetryClient.TrackEvent("CustomEvent", new Dictionary<string, string>
        {
            { "User", "TestUser" },
            { "Action", "MetricTracked" }
        });

        _logger.LogInformation("Metric tracked successfully");
        
        return Ok("Metric tracked");
    }

    [HttpGet("exception")]
    public IActionResult TrackException()
    {
        try
        {
            throw new InvalidOperationException("Test exception for Application Insights");
        }
        catch (Exception ex)
        {
            _telemetryClient.TrackException(ex);
            _logger.LogError(ex, "Exception tracked");
            return StatusCode(500, "Exception tracked");
        }
    }
}
```

#### 2.2 Kusto クエリの例（KQL）

Application Insights のログを分析：

```kql
// 過去24時間のリクエスト数
requests
| where timestamp > ago(24h)
| summarize RequestCount = count() by bin(timestamp, 1h)
| render timechart

// 失敗したリクエスト
requests
| where success == false
| where timestamp > ago(24h)
| project timestamp, name, resultCode, duration
| order by timestamp desc

// 例外の集計
exceptions
| where timestamp > ago(24h)
| summarize ExceptionCount = count() by type, outerMessage
| order by ExceptionCount desc

// カスタムメトリクス
customMetrics
| where name == "CustomMetric"
| where timestamp > ago(24h)
| summarize avg(value) by bin(timestamp, 5m)
| render timechart
```

#### 2.3 CLI でのクエリ実行（オプション）

Azure Portal の代わりに、Azure CLI を使用してKQLクエリを実行・保存することもできます。

##### 方法1: CLIでクエリを実行

```powershell
# リソースグループ名とアプリ名を変数化
$rgName = "rg-az400-handson3"
$appName = "az400-appinsights"

# 過去24時間のリクエスト数（テーブル形式）
az monitor app-insights query `
  --app $appName `
  --resource-group $rgName `
  --analytics-query "requests | where timestamp > ago(24h) | summarize RequestCount = count() by bin(timestamp, 1h)" `
  --output table

# 結果をJSONファイルに保存
az monitor app-insights query `
  --app $appName `
  --resource-group $rgName `
  --analytics-query "requests | where timestamp > ago(24h) | summarize RequestCount = count() by bin(timestamp, 1h)" `
  --output json > request-counts.json

# 失敗したリクエストを確認
az monitor app-insights query `
  --app $appName `
  --resource-group $rgName `
  --analytics-query "requests | where success == false | where timestamp > ago(24h) | project timestamp, name, resultCode, duration | order by timestamp desc" `
  --output table

# 例外の集計
az monitor app-insights query `
  --app $appName `
  --resource-group $rgName `
  --analytics-query "exceptions | where timestamp > ago(24h) | summarize ExceptionCount = count() by type, outerMessage | order by ExceptionCount desc" `
  --output table
```

##### 方法2: クエリをファイルに保存して実行

```powershell
# queriesフォルダーを作成
New-Item -ItemType Directory -Path "queries" -Force

# KQLクエリをファイルに保存
@"
requests
| where timestamp > ago(24h)
| summarize RequestCount = count() by bin(timestamp, 1h)
"@ | Out-File -FilePath "queries\request-counts.kql" -Encoding utf8

# ファイルから読み込んで実行
$query = Get-Content "queries\request-counts.kql" -Raw
az monitor app-insights query `
  --app $appName `
  --resource-group $rgName `
  --analytics-query $query `
  --output table
```

##### 方法3: 複数クエリの一括実行スクリプト

```powershell
# kql-queries.ps1
$rgName = "rg-az400-handson3"
$appName = "az400-appinsights"

# クエリ定義
$queries = @{
    "RequestCounts" = "requests | where timestamp > ago(24h) | summarize RequestCount = count() by bin(timestamp, 1h)"
    "FailedRequests" = "requests | where success == false | where timestamp > ago(24h) | project timestamp, name, resultCode, duration | order by timestamp desc"
    "Exceptions" = "exceptions | where timestamp > ago(24h) | summarize ExceptionCount = count() by type, outerMessage | order by ExceptionCount desc"
    "CustomMetrics" = "customMetrics | where name == 'CustomMetric' | where timestamp > ago(24h) | summarize avg(value) by bin(timestamp, 5m)"
}

# 結果保存用フォルダーを作成
New-Item -ItemType Directory -Path "results" -Force

# 各クエリを実行して結果を保存
foreach ($queryName in $queries.Keys) {
    Write-Host "Running query: $queryName" -ForegroundColor Cyan
    
    $result = az monitor app-insights query `
      --app $appName `
      --resource-group $rgName `
      --analytics-query $queries[$queryName] `
      --output json
    
    # 結果をファイルに保存
    $timestamp = Get-Date -Format 'yyyyMMdd-HHmmss'
    $result | Out-File -FilePath "results\$queryName-$timestamp.json"
    
    # テーブル形式でも表示
    az monitor app-insights query `
      --app $appName `
      --resource-group $rgName `
      --analytics-query $queries[$queryName] `
      --output table
    
    Write-Host ""
}

Write-Host "Query results saved to: results\" -ForegroundColor Green
```

> **💡 ヒント:** 定期的な監視レポート作成には、方法3のスクリプトをタスクスケジューラーやAzure Automationで自動実行すると便利です。

**Azure Monitor Agent (AMA) の役割:**

#### 6.1 Lead Time / Cycle Time / Velocity の理解

| メトリクス | 定義 | 用途 | 計算期間 |

| **Velocity** | スプリントで完了したポイント数 | 生産性・計画精度 | スプリント単位 |

#### 8.1 Pipeline 成功率の監視

```yaml
# azure-pipelines-metrics.yml
trigger:
  branches:
    include:
    - main

pool:
  vmImage: 'ubuntu-latest'

steps:
- script: |
    echo "Build started at: $(date)"
    echo "Build ID: $(Build.BuildId)"
    echo "Source Branch: $(Build.SourceBranch)"
  displayName: 'Log Build Info'

- task: PublishBuildArtifacts@1
  inputs:
    PathtoPublish: '$(Build.SourcesDirectory)'
    ArtifactName: 'metrics'

# Application Insights にメトリクスを送信（Connection String使用 - Azure推奨）
- script: |
    # Connection Stringから必要な情報を抽出
    CONNECTION_STRING="$(APP_INSIGHTS_CONNECTION_STRING)"
    
    # InstrumentationKeyを抽出
    IKEY=$(echo "$CONNECTION_STRING" | grep -oP 'InstrumentationKey=\K[^;]+')
    
    # IngestionEndpointを抽出（存在しない場合はデフォルトを使用）
    ENDPOINT=$(echo "$CONNECTION_STRING" | grep -oP 'IngestionEndpoint=\K[^;]+')
    if [ -z "$ENDPOINT" ]; then
      ENDPOINT="https://japaneast-1.in.applicationinsights.azure.com"
    fi
    
    # メトリクスを送信
    curl -X POST "${ENDPOINT}/v2.1/track" \
      -H "Content-Type: application/json" \
      -d "{
        \"name\": \"Microsoft.ApplicationInsights.Event\",
        \"time\": \"$(date -u +%Y-%m-%dT%H:%M:%S.%3NZ)\",
        \"iKey\": \"$IKEY\",
        \"data\": {
          \"baseType\": \"EventData\",
          \"baseData\": {
            \"name\": \"PipelineCompleted\",
            \"properties\": {
              \"BuildId\": \"$(Build.BuildId)\",
              \"Result\": \"Succeeded\"
            }
          }
        }
      }"
  displayName: 'Send Metrics to App Insights'
  env:
    APP_INSIGHTS_CONNECTION_STRING: $(AppInsightsConnectionString)
```

> **📌 重要な変更点（2026年推奨）:**
> - ❌ **非推奨**: `Instrumentation Key` 単体での使用

> - ✅ **推奨**: リージョン別エンドポイント（例: `japaneast-1.in.applicationinsights.azure.com`）
> - ✅ **推奨**: Azure DevOps変数を使用したセキュアな管理

   - **By value x-axis**: `1 hour`
5. グラフの棒をクリックして、イベント詳細を確認：
   - **Custom Properties** → `BuildId`
   - **Custom Properties** → `Result`

**方法2: Logs (KQL) で確認（より直接的）**

1. **Azure Portal** → **Application Insights** → `az400-appinsights`
2. 左側メニューの **「Monitoring」セクション** → **「Logs」** をクリック
3. 以下のKQLクエリを実行：
   ```kql
   customEvents
   | where name == "PipelineCompleted"
   | project timestamp, 
             BuildId = tostring(customDimensions.BuildId), 
             Result = tostring(customDimensions.Result)
   | order by timestamp desc
   | take 20
   ```
4. **「Run」** をクリック
5. 結果テーブルでBuildIdとResultを確認

> **💡 Logsを使うメリット:**
> - Eventsでフィルタ設定する必要がない
> - 複雑な条件でデータを抽出できる
> - 結果をテーブル形式で見やすく表示

**方法3: Analytics でトレンド分析**

```kql
customEvents
| where name == "PipelineCompleted"
| summarize Count = count() by bin(timestamp, 1h), Result = tostring(customDimensions.Result)
| render timechart
```

これで、パイプライン実行回数の時系列トレンドを可視化できます。

**トラブルシューティング:**

| 問題 | 原因 | 解決策 |
|------|------|--------|
| **Events に "PipelineCompleted" が表示されない** | パイプラインがまだ実行されていない、またはConnection Stringが未設定 | 1. パイプラインを実行<br>2. ステップ0でConnection String変数を設定<br>3. 数分待ってからRefresh |
| **"0 events" と表示される** | データ取得期間が短すぎる、またはイベントが送信されていない | 1. "During" を "Last 7 days" に変更<br>2. Logsで `customEvents` の全件確認<br>3. パイプラインログで curl のレスポンスを確認（HTTP 200が成功） |
| パイプラインが失敗する | YAMLの構文エラー | エラーメッセージを確認、インデントをチェック |
| Artifact が見つからない | パスが間違っている | `$(Build.SourcesDirectory)` を正しく指定 |
| **変数 AppInsightsConnectionString が見つからない** | ステップ0で変数を設定していない | Pipeline変数にConnection Stringを設定（ステップ0参照） |
| **IKEY や ENDPOINT が空** | Connection String形式が間違っている | Azure PortalからConnection Stringを正しくコピー |
| メトリクスが送信されない | エンドポイントまたはConnection Stringが間違っている | ステップ0の設定を確認 |

**【Option A】Application Insights送信ステップを削除（推奨）**

この演習の主目的は**パイプラインの統計情報確認**です。Application Insights へのメトリクス送信は**オプション**なので、削除しても問題ありません。

**修正後のYAML（シンプル版）:**
```yaml
# azure-pipelines-metrics.yml
trigger:
  branches:
    include:
    - main

pool:
  vmImage: 'ubuntu-latest'

steps:
- script: |
    echo "Build started at: $(date)"
    echo "Build ID: $(Build.BuildId)"
    echo "Source Branch: $(Build.SourceBranch)"
  displayName: 'Log Build Info'

- task: PublishBuildArtifacts@1
  inputs:
    PathtoPublish: '$(Build.SourcesDirectory)'
    ArtifactName: 'metrics'
  displayName: 'Publish Artifacts'
```

この設定で以下が確認できます：
- ✅ パイプラインの実行成功率
- ✅ ビルドログ（Build ID、Branch情報）
- ✅ Artifacts の発行
- ✅ Pipeline Analytics（成功率、実行時間のトレンド）

**【Option B】Azure CLIタスクを使用してApplication Insightsに送信**

Application Insightsへのメトリクス送信を行いたい場合は、`curl` の代わりに**Azure CLIタスク**を使用します：

```yaml
# Application Insights にメトリクスを送信（Azure CLI版）
- task: AzureCLI@2
  inputs:
    azureSubscription: 'your-service-connection-name'
    scriptType: 'bash'
    scriptLocation: 'inlineScript'
    inlineScript: |
      # カスタムイベントを送信
      az monitor app-insights events show \
        --app az400-appinsights \
        --resource-group rg-az400-handson3 \
        --type customEvents
      
      echo "Pipeline BuildId: $(Build.BuildId) completed successfully"
  displayName: 'Send Metrics to App Insights (Azure CLI)'
```

> **⚠️ 注意:** 
> - Azure CLI タスクを使用する場合、Service Connection の設定が必要です
> - `your-service-connection-name` は Azure DevOps の Project settings → Service connections で作成した接続名に置き換えてください

**【参考】Connection String vs Instrumentation Key の違い**

| 項目 | Instrumentation Key（非推奨） | Connection String（推奨） |
|------|-------------------------------|---------------------------|
| **含まれる情報** | キーのみ | キー + エンドポイント + その他設定 |
| **エンドポイント** | ハードコードが必要 | 自動的に含まれる |
| **リージョン対応** | 手動設定が必要 | 自動的に最適化 |
| **Azure推奨度** | ❌ 非推奨（将来廃止予定） | ✅ **推奨（2026年現在）** |
| **セキュリティ** | 低（キーのみ） | 高（複合情報） |

**Connection Stringの形式例:**
```
InstrumentationKey=26065dca-2103-4293-bb40-9652f9ee2a3b;
IngestionEndpoint=https://japaneast-1.in.applicationinsights.azure.com/;
LiveEndpoint=https://japaneast.livediagnostics.monitor.azure.com/;
ApplicationId=xxx-yyy-zzz
```

**上記YAMLでConnection Stringを使用する理由:**
- ✅ リージョン別のエンドポイントを自動抽出
- ✅ 将来の仕様変更に対応しやすい
- ✅ Azureのベストプラクティスに準拠

> **💡 実践的な使い方:** 
> - このパイプラインを定期的に実行（スケジュールトリガー）
> - 成功率を Analytics で追跡
> - 失敗時は Slack/Teams 通知と連携（Exercise 3-4 の応用）

- チャンネル名が正しいか確認（`#`を含む）
