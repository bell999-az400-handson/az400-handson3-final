#### 1.5 Client Secret を Key Vault に保存

**セキュリティベストプラクティス**: すべての機密情報は Key Vault で一元管理します。

**ステップ1: Key Vault へのアクセス権限を確認・付与**

```powershell
# 現在のユーザーを確認
$currentUser = az account show --query user.name -o tsv
Write-Host "Current user: $currentUser" -ForegroundColor Cyan

# オブジェクトIDを取得
$objectId = az ad signed-in-user show --query id -o tsv
Write-Host "Object ID: $objectId" -ForegroundColor Cyan

# リソースグループとサブスクリプション情報を取得
$rgName = "rg-az400-handson3"
$subscriptionId = az account show --query id -o tsv

# Key Vault 名を取得
$kvName = az keyvault list `
  --resource-group $rgName `
  --query "[0].name" -o tsv

Write-Host "Key Vault: $kvName" -ForegroundColor Yellow

# 現在のロール割り当てを確認
$existingRole = az role assignment list `
  --assignee $objectId `
  --scope "/subscriptions/$subscriptionId/resourceGroups/$rgName/providers/Microsoft.KeyVault/vaults/$kvName" `
  --query "[?roleDefinitionName=='Key Vault Secrets Officer' || roleDefinitionName=='Key Vault Administrator'].roleDefinitionName" -o tsv

if ($existingRole) {
    Write-Host "✅ Already has role: $existingRole" -ForegroundColor Green
} else {
    Write-Host "⚠️  No role assigned. Assigning 'Key Vault Secrets Officer' role..." -ForegroundColor Yellow
    
    # Key Vault Secrets Officer ロールを割り当て（読み書き可能）
    az role assignment create `
      --role "Key Vault Secrets Officer" `
      --assignee-object-id $objectId `
      --scope "/subscriptions/$subscriptionId/resourceGroups/$rgName/providers/Microsoft.KeyVault/vaults/$kvName"
    
    Write-Host "✅ Role assigned. Waiting for RBAC propagation (30 seconds)..." -ForegroundColor Green
    Start-Sleep -Seconds 30
}
```

**ステップ2: Key Vault にシークレットを保存**

```powershell
# Key Vault 名を設定（Lab 2 で作成したものを使用）
# 既存の Key Vault 名を確認する場合:
# az keyvault list --query "[].name" -o table

# 上記で取得済みの場合はスキップ
# $kvName = az keyvault list `
#   --resource-group "rg-az400-handson3" `
#   --query "[0].name" -o tsv

Write-Host "Using Key Vault: $kvName" -ForegroundColor Yellow

# Service Principal の値を変数に保存（1.1 の出力から取得）
$spAppId = Read-Host -Prompt "Application ID (appId) を入力"
$spPassword = Read-Host -Prompt "Client Secret (password) を入力" -AsSecureString | ConvertFrom-SecureString -AsPlainText
$spTenantId = Read-Host -Prompt "Tenant ID (tenant) を入力"

# Key Vault にシークレットを保存
az keyvault secret set `
  --vault-name $kvName `
  --name "sp-app-id" `
  --value $spAppId

az keyvault secret set `
  --vault-name $kvName `
  --name "sp-client-secret" `
  --value $spPassword

az keyvault secret set `
  --vault-name $kvName `
  --name "sp-tenant-id" `
  --value $spTenantId

Write-Host "✅ Service Principal credentials saved to Key Vault: $kvName" -ForegroundColor Green

# 保存されたシークレットを確認
az keyvault secret list --vault-name $kvName --query "[?starts_with(name, 'sp-')].{Name:name, Enabled:attributes.enabled}" -o table
```

**メリット**:
- ✅ シークレットの一元管理
- ✅ RBAC によるアクセス制御
- ✅ 監査ログでアクセス追跡
- ✅ シークレットのローテーションが容易
- ✅ 環境変数にシークレットを保存しない（セキュリティリスク低減）

### Exercise 2: Service Principal でログイン

#### 2.1 Key Vault からシークレットを取得

**セキュリティベストプラクティス**: 環境変数に直接シークレットを設定せず、Key Vault から動的に取得します。

```powershell
# リソースグループとKey Vault名を設定
$rgName = "rg-az400-handson3"
$kvName = az keyvault list `
  --resource-group $rgName `
  --query "[0].name" -o tsv

Write-Host "Retrieving credentials from Key Vault: $kvName" -ForegroundColor Cyan

# アクセス権限を確認（読み取り専用の場合は Key Vault Secrets User で十分）
$objectId = az ad signed-in-user show --query id -o tsv
$subscriptionId = az account show --query id -o tsv

$hasReadAccess = az role assignment list `
  --assignee $objectId `
  --scope "/subscriptions/$subscriptionId/resourceGroups/$rgName/providers/Microsoft.KeyVault/vaults/$kvName" `
  --query "[?roleDefinitionName=='Key Vault Secrets User' || roleDefinitionName=='Key Vault Secrets Officer' || roleDefinitionName=='Key Vault Administrator'].roleDefinitionName" -o tsv

if (-not $hasReadAccess) {
    Write-Host "⚠️  No read access. Assigning 'Key Vault Secrets User' role..." -ForegroundColor Yellow
    
    az role assignment create `
      --role "Key Vault Secrets User" `
      --assignee-object-id $objectId `
      --scope "/subscriptions/$subscriptionId/resourceGroups/$rgName/providers/Microsoft.KeyVault/vaults/$kvName"
    
    Write-Host "✅ Role assigned. Waiting for RBAC propagation (30 seconds)..." -ForegroundColor Green
    Start-Sleep -Seconds 30
}

# Key Vault からシークレットを取得
$spAppId = az keyvault secret show `
  --vault-name $kvName `
  --name "sp-app-id" `
  --query value -o tsv

$spPassword = az keyvault secret show `
  --vault-name $kvName `
  --name "sp-client-secret" `
  --query value -o tsv

$spTenantId = az keyvault secret show `
  --vault-name $kvName `
  --name "sp-tenant-id" `
  --query value -o tsv

# 確認（値は表示しない）
if ($spAppId -and $spPassword -and $spTenantId) {
    Write-Host "✅ Successfully retrieved all credentials from Key Vault" -ForegroundColor Green
} else {
    Write-Host "❌ Failed to retrieve credentials" -ForegroundColor Red
}
```

#### 3.2 Service Connection の作成

**3つの作成方法の比較:**

| 方法 | 自動化レベル | Client Secret | 難易度 | 推奨シーン |
|------|--------------|---------------|--------|------------|
| 方法1: Azure DevOps CLI | 部分的 | 手動更新が必要 | 中 | CLI に慣れている場合 |
| 方法2: REST API | 完全自動化 | ✅ 自動設定 | 高 | CI/CD で完全自動化したい場合 |
| 方法3: ポータル | 手動 | ✅ GUI で設定 | 低 | 初めて作成する場合、確認しながら作業 |

**方法1: スクリプトで自動作成（Azure DevOps CLI）**

```powershell
# Azure DevOps 拡張機能をインストール（初回のみ）
az extension add --name azure-devops

# Azure DevOps の組織とプロジェクトを設定
$org = "https://dev.azure.com/bell999"  # 自分の組織URLに変更
$project = "az400-handson3"  # 自分のプロジェクト名に変更

# Key Vault 名を先に取得
$kvName = az keyvault list --resource-group "rg-az400-handson3" --query "[0].name" -o tsv
Write-Host "Using Key Vault: $kvName" -ForegroundColor Cyan

# Azure DevOps にログイン（PATを使用）
# 💡 PATがすでにKey Vaultに保存されている場合は、こちらを使用:
$env:AZURE_DEVOPS_EXT_PAT = az keyvault secret show --vault-name $kvName --name "azure-devops-pat" --query value -o tsv

# 💡 PATをKey Vaultに保存していない場合は、手動入力:
# $env:AZURE_DEVOPS_EXT_PAT = Read-Host -Prompt "Azure DevOps PAT を入力" -AsSecureString | ConvertFrom-SecureString -AsPlainText

# デフォルトの組織とプロジェクトを設定
az devops configure --defaults organization=$org project=$project

# Key Vault から Service Principal の値を取得
$spAppId = az keyvault secret show --vault-name $kvName --name "sp-app-id" --query value -o tsv
$spPassword = az keyvault secret show --vault-name $kvName --name "sp-client-secret" --query value -o tsv
$spTenantId = az keyvault secret show --vault-name $kvName --name "sp-tenant-id" --query value -o tsv
$subscriptionId = az account show --query id -o tsv
$subscriptionName = az account show --query name -o tsv

# 💡 Client Secret をクリップボードにコピー（次のコマンドで入力を求められます）
$spPassword | Set-Clipboard
Write-Host "✅ Client Secret をクリップボードにコピーしました（Ctrl+V で貼り付け可能）" -ForegroundColor Green

# Service Connection を作成
# 💡 実行すると「Azure RM service principal key:」と聞かれます
# → Ctrl+V で $spPassword（Client Secret）を貼り付けてください
az devops service-endpoint azurerm create `
  --azure-rm-service-principal-id $spAppId `
  --azure-rm-subscription-id $subscriptionId `
  --azure-rm-subscription-name $subscriptionName `
  --azure-rm-tenant-id $spTenantId `
  --name "AzureServiceConnection" `
  --org $org `
  --project $project

# Service Connection ID を取得
$serviceEndpointId = az devops service-endpoint list `
  --org $org `
  --project $project `
  --query "[?name=='AzureServiceConnection'].id" -o tsv

# Service Principal Key（Client Secret）を更新
az devops service-endpoint update `
  --id $serviceEndpointId `
  --enable-for-all true `
  --org $org `
  --project $project

Write-Host "✅ Service Connection created successfully: AzureServiceConnection" -ForegroundColor Green

# 確認
az devops service-endpoint list `
  --org $org `
  --project $project `
  --query "[?name=='AzureServiceConnection'].{Name:name, Type:type, IsReady:isReady}" -o table
```

**💡 注意**: `az devops service-endpoint azurerm create` コマンドではClient Secretを直接設定できないため、作成後に Azure DevOps ポータルで一度Secretを更新する必要があります。または、方法2の完全自動化スクリプトを使用してください。

**⚠️ トラブルシューティング:**

エラー: `TF400813: The user 'aaaaaaaa-aaaa-aaaa-aaaa-aaaaaaaaaaaa' is not authorized to access this resource.`

**原因1: Service Principalでログインしたまま**
```powershell
# ユーザータイプを確認
az account show --query user.type -o tsv

# "servicePrincipal" の場合は通常ユーザーに戻す
```

**原因2: PATが未設定または期限切れ**
```powershell
# 環境変数を確認
$env:AZURE_DEVOPS_EXT_PAT

# 未設定の場合は設定

```

**原因3: Azure DevOpsプロジェクトへのアクセス権限不足**
- Azure DevOps → Project Settings → Permissions で、自分のアカウントに **Project Administrators** または **Endpoint Administrators** ロールがあるか確認

**方法2: REST API で完全自動作成（Client Secret含む）**

```powershell
# 前提: 方法1と同じ変数を設定済み
# $org, $project, $kvName, $spAppId, $spPassword, $spTenantId, $subscriptionId, $subscriptionName

# Azure DevOps PAT を取得
$pat = $env:AZURE_DEVOPS_EXT_PAT
if (-not $pat) {
    $pat = Read-Host -Prompt "Azure DevOps PAT を入力" -AsSecureString | ConvertFrom-SecureString -AsPlainText
}

# Base64 エンコードされた認証ヘッダーを作成
$encodedPat = [Convert]::ToBase64String([Text.Encoding]::ASCII.GetBytes(":$pat"))

# プロジェクトIDを取得
$projectUrl = "$org/_apis/projects/$project`?api-version=7.0"
$headers = @{
    Authorization = "Basic $encodedPat"
    "Content-Type" = "application/json"
}
$projectInfo = Invoke-RestMethod -Uri $projectUrl -Headers $headers -Method Get
$projectId = $projectInfo.id

# Service Connection の JSON ペイロードを作成
$body = @{
    data = @{
        subscriptionId = $subscriptionId
        subscriptionName = $subscriptionName
        environment = "AzureCloud"
        scopeLevel = "Subscription"
        creationMode = "Manual"
    }
    name = "AzureServiceConnection"
    type = "azurerm"
    url = "https://management.azure.com/"
    authorization = @{
        parameters = @{
            serviceprincipalid = $spAppId
            authenticationType = "spnKey"
            serviceprincipalkey = $spPassword
            tenantid = $spTenantId
        }
        scheme = "ServicePrincipal"
    }
    isShared = $false
    isReady = $true
    serviceEndpointProjectReferences = @(
        @{
            projectReference = @{
                id = $projectId
                name = $project
            }
            name = "AzureServiceConnection"
        }
    )
} | ConvertTo-Json -Depth 10

# Service Connection を作成
$serviceEndpointUrl = "$org/$project/_apis/serviceendpoint/endpoints?api-version=7.0"
try {
    $response = Invoke-RestMethod -Uri $serviceEndpointUrl -Headers $headers -Method Post -Body $body
    
    # 作成されたService Connectionを確認（レスポンスが空の場合があるため）
    Start-Sleep -Seconds 2
    $listUrl = "$org/$project/_apis/serviceendpoint/endpoints?api-version=7.0"
    $endpoints = Invoke-RestMethod -Uri $listUrl -Headers $headers -Method Get
    $createdConnection = $endpoints.value | Where-Object { $_.name -eq "AzureServiceConnection" }
    
    if ($createdConnection) {
        Write-Host "✅ Service Connection created successfully!" -ForegroundColor Green
        Write-Host "   Name: $($createdConnection.name)" -ForegroundColor Cyan
        Write-Host "   ID: $($createdConnection.id)" -ForegroundColor Cyan
        Write-Host "   Type: $($createdConnection.type)" -ForegroundColor Cyan
        
        $endpointId = $createdConnection.id
        
        # すべてのパイプラインからアクセス可能にする
        Write-Host "`nGranting access to all pipelines..." -ForegroundColor Yellow
        
        # 注意: Pipeline permissions API は認証エラーが発生する場合があります
        # その場合は手動で設定してください
        $pipelinePermissionUrl = "$org/_apis/pipelines/pipelinePermissions/endpoint/$endpointId`?api-version=7.1-preview.1"
        $permissionBody = @{
            pipelines = @()
            allPipelines = @{
                authorized = $true
            }
        } | ConvertTo-Json -Depth 10
        
        try {
            Invoke-RestMethod -Uri $pipelinePermissionUrl -Headers $headers -Method Patch -Body $permissionBody
            Write-Host "✅ Granted access to all pipelines" -ForegroundColor Green
        } catch {
            Write-Host "⚠️  Could not set pipeline permissions via API" -ForegroundColor Yellow
            Write-Host "   Error: $($_.Exception.Message)" -ForegroundColor Gray
            Write-Host "`n📌 Please grant permissions manually:" -ForegroundColor Cyan
            Write-Host "   1. Go to: $org/$project/_settings/adminservices" -ForegroundColor Gray
            Write-Host "   2. Find 'AzureServiceConnection'" -ForegroundColor Gray
            Write-Host "   3. Click '...' > Security" -ForegroundColor Gray
            Write-Host "   4. Enable 'Grant access permission to all pipelines'" -ForegroundColor Gray
        }
    } else {
        Write-Host "⚠️  Service Connection created but not found in list" -ForegroundColor Yellow
        Write-Host "   Please check manually: $org/$project/_settings/adminservices" -ForegroundColor Gray
    }
    
} catch {
    Write-Host "❌ Failed to create Service Connection" -ForegroundColor Red
    Write-Host "Error: $($_.Exception.Message)" -ForegroundColor Red
    Write-Host "Status Code: $($_.Exception.Response.StatusCode.value__)" -ForegroundColor Red
}
```

- **手動設定が必要**: Pipeline permissionsの設定でエラーが出た場合は、Azure DevOps UIで手動設定してください

**⚠️ Azure Reposでは:**

- Status Checkに「SonarCloud/quality gate」が表示されなくても問題なし
- GitHub連携時のみ「SonarCloud/quality gate」が表示される

**🎓 AZ-400試験のポイント:**
- Azure Repos: Build Validationを使用
- GitHub連携: Status Check（GitHub Checks API）を使用
- 目的は同じ：コード品質基準を満たさないPRのマージをブロック

##### T3: Checksセクションに何も表示されない {#ts-5-3-no-checks}

**問題:**
- PRを作成したが、Checksセクション自体が表示されない
- または「SonarQube」が表示されない

**診断手順:**

**ステップ1: パイプライン実行を確認**
```
Azure DevOps → Pipelines → Pipelines
→ 最新の実行履歴を確認
→ PR作成後にパイプラインが実行されたか？
```
- 実行されていない → [T2: PRで実行されない](#ts-5-3-pr-trigger)へ
- 実行されたが待機中 → [T4: Permitで止まる](#ts-5-3-permit)へ
- 実行されたが失敗 → ログを確認して原因を特定

**ステップ2: PRコンテキストで実行されたか確認**

パイプライン実行ログの冒頭で以下を確認：
```yaml
Variables:
  Build.Reason: PullRequest  # ← これが表示されているか？
  System.PullRequest.PullRequestId: 123  # ← PR番号が表示されているか？
```
- `Build.Reason: Manual` の場合 → 手動実行ではChecksに表示されません。PRを更新して自動実行させてください
- `Build.Reason: PullRequest` の場合 → 正しくPRコンテキストで実行されています

**ステップ3: パイプライン完了を待つ**
- パイプラインが完了するまで数分かかります
- 完了後、PRページをリフレッシュ（Ctrl+F5）してください

##### T4: パイプラインが待機状態で止まる（Permit） {#ts-5-3-permit}

2. リソース一覧が表示された場合、すべてにチェックを入れて「**Permit**」をクリック

- リポジトリへのアクセス
- エージェントプールの使用

**⚠️ 重要:**
- Permit しないとパイプラインは永久に待機状態のままです
- 初回実行時は必ず確認してください
- 2回目以降は自動的に許可されます

##### T6: SonarCloud警告の修正方法 {#ts-5-3-warnings}

**問題:**
- パイプラインは成功するが、SonarCloudで警告が表示される
- 例: S3903, S6966, S2699

**理解:**
これらは**Code Smells**（コード品質改善提案）であり、Quality Gateの**失敗ではありません**。

**デフォルトのQuality Gate基準:**
- 新しいコードのバグ: 0
- 新しいコードの脆弱性: 0
- 新しいコードのカバレッジ: 80%以上
- 新しいコードの重複: 3%以下

**警告は出ますが、Quality Gateは通過します**（重大な問題ではないため）。

**📝 オプション：警告を修正してコード品質を向上**

<details>
<summary><b>警告の修正例（クリックして展開）</b></summary>

#### 6.2 Key Vault へのアクセス権限を付与

**⚠️ 重要:** Key VaultがRBACモデル（`--enable-rbac-authorization`）で作成されている場合、`set-policy`コマンドは使用できません。代わりにRBACロールを割り当てます。

```powershell
# Key Vault 名を設定（既存のKey Vaultを使用）
$rgName = "rg-az400-handson3"
$kvName = az keyvault list `
  --resource-group $rgName `
  --query "[0].name" -o tsv

# サブスクリプションIDを取得
$subscriptionId = az account show --query id -o tsv

# Managed Identity に Key Vault の読み取り権限を付与（RBAC方式）
# "Key Vault Secrets User" ロールを割り当て（読み取り専用）
az role assignment create `
  --role "Key Vault Secrets User" `
  --assignee-object-id $principalId `
  --scope "/subscriptions/$subscriptionId/resourceGroups/$rgName/providers/Microsoft.KeyVault/vaults/$kvName"

Write-Host "✅ Managed Identity に Key Vault Secrets User ロールを付与しました" -ForegroundColor Green

# ロール割り当てを確認
az role assignment list `
  --assignee $principalId `
  --scope "/subscriptions/$subscriptionId/resourceGroups/$rgName/providers/Microsoft.KeyVault/vaults/$kvName" `
  --query "[].{Role:roleDefinitionName, Scope:scope}" -o table
```

**💡 補足:**
- **Key Vault Secrets User**: シークレットの読み取り専用（推奨）
- **Key Vault Secrets Officer**: シークレットの読み書き可能
- RBACの変更は反映に最大30秒かかる場合があります

## 🔍 トラブルシューティング

### Azure DevOps PAT のスコープ不足

**症状**: REST API で Service Connection 作成時に 401 Unauthorized エラー

**原因**: Personal Access Token (PAT) に必要なスコープ（権限）が不足

**解決方法**:

1. Azure DevOps → User settings (右上のアイコン) → Personal access tokens
2. 既存のPATを選択または新規作成
3. 必要なスコープを確認:
   - ✅ **Service Connections**: `Read, Query, & Manage` (必須)
   - ✅ **Build**: `Read & execute` (Pipeline permissions用)
   - ✅ **Project and Team**: `Read, write, & manage`
   - ✅ **Code**: `Read` (任意)
4. PATをコピーして環境変数に設定:

```powershell
# PATを環境変数に設定
$env:AZURE_DEVOPS_EXT_PAT = "your-pat-token-here"

# または、Key Vaultに保存（推奨）
az keyvault secret set `
  --vault-name "kv-az400-bell9" `
  --name "azure-devops-pat" `
  --value "your-pat-token-here"

# Key Vaultから取得
$pat = az keyvault secret show `
  --vault-name "kv-az400-bell9" `
  --name "azure-devops-pat" `
  --query value -o tsv
```

### Key Vault にアクセスできない（Forbidden エラー）

**エラー例:**
```
ERROR: (Forbidden) Caller is not authorized to perform action on resource.
Action: 'Microsoft.KeyVault/vaults/secrets/getSecret/action'
Inner error: { "code": "ForbiddenByRbac" }
```

**原因**: Key Vault が RBAC モデルを使用しており、現在のユーザーにシークレット読み取り権限がない

**解決方法:**

```powershell
# 現在のユーザーを確認
az account show --query user.name -o tsv

# オブジェクトIDを取得
$objectId = az ad signed-in-user show --query id -o tsv

# Key Vault の情報を設定
$kvName = "kv-az400-bell9"  # 実際の Key Vault 名に置き換え
$rgName = "rg-az400-handson3"
$subscriptionId = az account show --query id -o tsv

# Key Vault Secrets User ロールを割り当て
az role assignment create `
  --role "Key Vault Secrets User" `
  --assignee-object-id $objectId `
  --scope "/subscriptions/$subscriptionId/resourceGroups/$rgName/providers/Microsoft.KeyVault/vaults/$kvName"

Write-Host "✅ Role assigned. Waiting for RBAC propagation..." -ForegroundColor Green
Start-Sleep -Seconds 30

# 再度シークレットを取得
az keyvault secret show --vault-name $kvName --name "sp-tenant-id" --query value -o tsv
```

**Service Principal の場合:**

```powershell
# Service Principal のアプリケーションIDを指定
$spAppId = "{your-service-principal-app-id}"

# Service Principal に権限を付与
az role assignment create `
  --role "Key Vault Secrets User" `
  --assignee $spAppId `
  --scope "/subscriptions/$subscriptionId/resourceGroups/$rgName/providers/Microsoft.KeyVault/vaults/$kvName"
```

**利用可能なロール:**

| ロール | 権限 | 用途 |
|--------|------|------|
| Key Vault Secrets User | シークレットの読み取りのみ | アプリケーション、パイプライン |
| Key Vault Secrets Officer | シークレットの読み書き | 管理者、DevOps エンジニア |
| Key Vault Administrator | すべての操作 | Key Vault 管理者 |

**💡 ヒント:**
- RBAC の変更は反映に最大10分かかる場合があります
- `--assignee-object-id` はオブジェクトIDを指定
- `--assignee` はアプリケーションID (Service Principal の場合) またはUPN (ユーザーの場合) を指定
